The Project: VeriCode

Every programming problem has three parts that can each go wrong on their own: the code that solves it, the tests that check it, and the specification that says what "correct" even means. This week you'll build all three, and on Day 4 you'll go one step further than testing alone can: getting an actual computer-checked proof, using a real research tool called Verus. Along the way, you'll use AI assistance at every stage: writing code, writing tests, writing specs, and even writing proofs, and see for yourself how much of each one it can actually do.

What is VeriCode?

VeriCode is the tool you'll build this week for checking whether code is actually correct, combining two different approaches: automated testing and formal verification. You'll build the testing half first (in Python on Day 2, then in Rust from Day 3 onward), then extend it into a real, computer-checked proof starting Day 4.

The testing half works like this. You give it:

  • a programming problem (we'll use "is this year a leap year?" and "is this number prime?"),
  • a table of test cases (input → expected answer),
  • and any candidate function: yours, a buggy one, or one an AI wrote.

VeriCode runs the candidate against every test case and reports exactly which ones pass and which ones fail. That's the same idea behind real automatic grading systems, and behind VeriContest, which is what inspired this internship.

But testing only ever tells you about the inputs you actually tried. On Day 4, you'll write a formal specification and use Verus (a real verification tool for Rust, the same kind of tool used in the research behind VeriContest) to get a mathematical guarantee that your code is correct for every possible input, not just the ones in your test table. That's VeriCode's other half: not just checking code against examples, but proving it.

Throughout the week, you'll also turn each of these steps over to an AI assistant and see how it does: does it write code that passes your tests? Does it write a spec that matches yours? Does it write a proof that actually verifies, including the tricky details Verus needs? By Day 5, you'll run one final problem through both the testing pipeline and the spec-and-proof pipeline yourself, using AI to help at every step, and compare what each one gives you.

Why tiers instead of one big spec? Finishing Tier 1 alone is a complete, real project. Everything after that makes it better, but nothing after Tier 1 is required for the week to have been a success. If you ever finish faster than expected, the Extensions page always has a next step ready. You'll never run out of things to build.
A heads-up: Rust and Verus have more to learn upfront than a language like Python would, which is exactly why Day 2 stays in Python and Day 3 is dedicated entirely to learning Rust before Day 4 asks you to write specs and proofs in it. Every code example on this site is also given to you working and ready to copy into the playground, so you're reading and adapting real code, not writing it from a blank page.
