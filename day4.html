Writing Formal Specifications, and How Much an AI Can Help

Today's goal: get comfortable turning a plain-English description into a precise, formal spec fn in Verus, and see how much of that an AI assistant can do for you. Today, you only ever write specs, never executable code: every exec function you see today is already given to you, complete and correct. Proofs (actually getting Verus to verify code matches a spec) are Day 5's job.

Do · first, before reading further

🤔 Ambiguity Quiz

Here's a problem: "count the words in a sentence." Sounds simple. For each input below, type the number of words you think it has, then reveal how others might disagree.

Most people say 2. But if you split naively on single spaces, you might get several empty "words" from the doubled-up spaces. A precise spec has to say exactly how to handle extra whitespace.
Most people say 1: it's one word with an apostrophe in it. But a program that splits text apart wherever there's punctuation would see two pieces, don and t, and might report 2. The English description never said how to treat apostrophes.
Is the answer 0? Or does a naive "split on spaces" approach report 1 (treating the empty string as a single empty "word")? This is exactly the kind of edge case a spec needs to pin down.
Is a hyphenated phrase 1 word or 3? Dictionaries disagree with each other on this in real life. There's no universally "correct" answer, only a chosen one.
If your answers differ from a partner's (or you could imagine someone reasonably disagreeing with your own answer), that's the point: the description was ambiguous. Removing that ambiguity, precisely, is exactly what today is about. Keep this in mind today: an AI reading an ambiguous English description faces exactly the same problem you just did.

📖 Introducing Verus

Verus is an open-source research tool for writing formally verified code in Rust. "Formal verification" means using mathematical logic to prove a program does what it's supposed to do, for every possible input, rather than testing a sample of inputs and hoping the rest behave the same way. That's a fundamentally stronger guarantee than testing: a test suite can only ever tell you "this passed on the inputs I tried"; a formal proof tells you "this is correct for every input," the same way a mathematical proof holds for every number, not just the ones anyone bothered to check.

Here's how it actually works. You write ordinary Rust code, then add extra annotations, like requires, ensures, and spec fn, describing precisely what your code promises to guarantee. Verus reads those annotations and tries to mathematically prove your code lives up to them, using an automatic theorem prover called an SMT solver (Verus uses one called Z3) to do the heavy logical lifting. Succeed, and you get a genuine mathematical guarantee. Fail, and Verus tells you exactly which promise it couldn't verify, the same way the Rust compiler points at exactly which line has a type error. Today, you're focused entirely on the first half of that picture: writing the promises (specs) precisely. Day 5 is where you'll spend more time on the second half: getting Verus to actually prove code lives up to them.

This is also why a tool like this targets Rust specifically, rather than a looser language like Python. Rust's own design (its strict type system, and its rules about who owns each piece of memory) already does a lot of the disciplined bookkeeping a verifier needs, giving Verus a running start. Verified Rust code is also still ordinary, fast, real Rust once it compiles: formal verification happens entirely before your program ever runs, adding a mathematical guarantee on top without changing what the compiled program is.

This is the actual research frontier VeriContest sits on, and the reason this whole internship uses Rust and Verus instead of a friendlier-looking language: you're using the real tool, not a simplified stand-in for it.

📖 What is a specification?

A specification states, precisely, what the correct answer is for every possible input, not just the examples in your test table. For leap years, our spec is actually already precise: "divisible by 4, except centuries (divisible by 100), which are not leap years, except centuries divisible by 400, which are leap years after all." The reason it's a good spec is that there's no reasonable way to read it two different ways.

      A spec fn body has to be purely mathematical: no

Saying "for every" and "there exists": forall and exists

    

      Some specs need to talk about every possible value of something, or
      claim that at least one value exists with some property. Verus has two
      keywords for exactly this, and you already saw forall once, in Day
      1's periodicity example. The syntax for both:
    

    

      
      

        Verus (spec)
      

      

    

    

      Read the pattern piece by piece: |x: int| introduces a new
      variable named x, of type int, that the rest of the
      expression is "for every" or "there exists" a value of. For forall,
      what follows is almost always an implication (==>): "for every
      x, IF some condition holds, THEN some conclusion follows." For
      exists, what follows is usually just a condition to satisfy, no
      implication needed. The #[trigger] annotation is a hint telling
      Verus which expression to watch for; you'll see it on almost every
      forall/exists you write, including in several
      exercises below.

📖 What is executable code?

    

      A spec by itself doesn't do anything: it can't be called from a
      program, can't run, can't produce an answer for a real input. The part that
      actually runs is what Verus calls executable code, or an
      exec fn: an ordinary Rust function, exactly like every
      is_leap, is_prime, and judge you've
      written this week. The whole point of this week's project is connecting the
      two: a spec that says what should be true, and executable code that
      actually computes an answer. Every exec function on this page from here on is
      already given to you, complete; today is only about writing the spec half of
      that connection.
    

    

      That connection is an ensures clause. It says: "I promise the
      result of this function always matches the spec," and Verus will refuse to
      accept the code unless it can prove that promise true for every

A trap: "verified" doesn't mean your spec is right

    

      There's a genuinely important limit to what "verified" tells you. Verus only
      checks that your executable code matches whatever spec you actually
      wrote. It has no idea what the problem was really supposed to be; if your
      spec is too weak, too vague, or accidentally says almost nothing at all,
      completely wrong code can still pass. Watch this:
    

    

      
      

        Verus
        
        
        Open Playground ↗
        
      

      
      

    

    

      Run it: you'll see verification results:: 2 verified, 0 errors,
      for a function that claims every single year is a leap year. Nothing
      about that code is correct. But is_leap_spec_trivial never
      actually says anything about the year at all, it just says true
      no matter what; so literally any executable code, right or wrong, satisfies
      it. Verus did exactly what it promised (proved the code matches the spec);
      the spec itself was simply empty, and nobody told Verus that was a problem,
      because it can't tell.
    

    

      The habit this should build: a green verified
      is the finish line for a proof-writing exercise, but not for a
      spec-writing one. Every time you finish filling in a spec today and
      it reports verified, that's your cue to click the reveal button
      and compare your spec, clause by clause, against the real one, not to stop.
      Ask yourself: does mine actually rule out every wrong answer the way the
      real one does, or does it just happen to survive against the one correct
      implementation sitting in the playground? An incomplete or overly loose spec
      can slip through unnoticed far more easily than incomplete code can, exactly
      because nothing about writing a spec forces you to think through every case
      the way writing a loop does. This is also a real, known danger of using AI
      to write specs: a plausible-looking but too-weak spec can pass verification
      and still be a bad spec.

🔨 7. Contains Duplicate: a spec straight from VeriContest

    

      Here's a genuine problem, completely unmodified, taken directly from the
      VeriContest
      dataset (946 real competitive-programming problems, each with a formal spec
      and a Verus proof): LeetCode 217, "Contains Duplicate."
    

    

      
      

        Real VeriContest problem description (lc217)
      

      

    

    

      This one genuinely needs exists, not just forall:
      "contains a duplicate" means "there exists some pair of positions holding
      the same value," which has no way to be phrased truthfully without it. You'll
      also spot forall in the requires clause below,
      bounding every value in the array; between the two, this problem uses both
      quantifiers for real. One more new piece of syntax in the code: you'll see
      HashSet, Rust's name for a set data structure, used here to
      remember "which values have I already seen." You don't need to understand
      it today, just recognize what it's doing.
    

    

      This problem's ensures clause is contains_duplicate_spec(nums@,
      res), not res == contains_duplicate_spec(...) like most of
      today's other problems. That difference matters, exactly because of the trap you
      just saw above: if you write a spec that ignores res and always
      comes out true (or anything logically equivalent to always-true,
      even if it doesn't look like it at a glance), it'll verify no
      matter what, without actually saying anything about duplicates. This is the
      single easiest problem today to accidentally pass without being correct, so
      comparing against the real spec afterward matters here more than anywhere else
      on this page.
