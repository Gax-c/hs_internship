Extensions: The Eight Queens Problem

Finished a day's core work early, or want a real coding challenge to take home? Here's a single classic computer science puzzle, worth going deep on rather than rushing through: the Eight Queens Problem. This page builds up everything you need from scratch, including two new ideas (depth-first search and backtracking) that you haven't seen yet this week, before asking you to write your own solution in Python.

Learn

📖 What is the Eight Queens Problem?

A chess queen can attack any piece in the same row, the same column, or along either diagonal from where it stands, as far as it can see in any of those 8 directions. The puzzle: place 8 queens on a standard 8×8 chessboard so that no two of them attack each other. No two queens may share a row, a column, or a diagonal, all eight at once.

Here's one valid arrangement (each Q is a queen; every other square is empty):

One valid 8-queens solution 

    

    

      Check a few pairs yourself: the queen in row 1 (column 1) and the queen in
      row 2 (column 5) don't share a row, column, or diagonal, so they're safe
      together. Now compare row 1's queen to row 5's queen (column 3): row
      difference is 4, column difference is 2, different, so no diagonal either.
      Every one of the 8C2 = 28 pairs in this board works
      out that way; that's what makes it a valid solution.
    

    

      One detail worth spelling out, since it's easy to picture wrong at first:
      two squares are on the same diagonal exactly when the number of rows
      between them equals the number of columns between them. A queen at row 2,
      column 2 and a queen at row 5, column 5 are 3 rows apart and 3 columns
      apart (3 == 3), so they share a diagonal. A queen at row 2, column 2 and
      one at row 5, column 4 are 3 rows apart but only 2 columns apart (3 != 2),
      so they don't. You'll use exactly this rule in the code below.
    

    

      It's harder than it looks by hand. There are 88, over 16
      million, ways to place 8 queens with one per column, and only
      92 of them (12, if you consider rotations and reflections
      of the same arrangement as "the same solution") actually work. Checking
      all 16 million by brute force is wasteful. The idea that makes it
      practical, backtracking, is a specific way of using
      depth-first search; both are genuinely useful ideas well
      beyond this one puzzle, so let's build them up properly before writing any
      queens-specific code.

📖 What is Depth-First Search (DFS)?

    

      Imagine exploring a cave with branching paths. At the very first fork, you
      can go LEFT or RIGHT. Whichever you pick
      leads to another fork, with the same two choices, and then another. If you
      always fully explore everything down one direction before ever trying the
      other, that's depth-first search: go as deep as you
      possibly can along one path first, and only try a different path once
      you've completely finished with the current one.

      branch before moving to the next, is the "depth-first" part of the name.
      Notice how it works: explore calls itself, once for
      each choice, with a slightly longer path each time; that's called
      recursion, and it's what makes exploring "try every
      possibility from here" so short to write.

📖 What is backtracking?

    

      DFS by itself just explores everything. Backtracking is
      DFS with one extra habit: after you make a choice and fully explore
      everything that follows from it, you undo that choice before
      trying the next one, so every new attempt starts from a clean slate. Three
      steps, every time: choose something, explore
      what happens next, then un-choose it again.

      Run it: you'll see all 6 orderings. The chosen.pop() line at
      the end is the "un-choose" step, and it's the entire reason this works:
      without it, the chosen list from the "red" branch would still
      have "red" stuck in it when the loop moves on to try "blue" first instead,
      and every later answer would come out wrong.
    

    

      For the Eight Queens problem, the same three-step pattern applies, with
      one addition: before you even bother exploring a choice, you first check
      whether it's safe (doesn't conflict with a queen already placed).
      If it isn't safe, skip it immediately, no exploring, no undoing needed,
      you just never chose it in the first place. That single check is what
      turns "try all 16 million placements" into "only ever explore placements
      that could possibly work," which is the entire reason backtracking is fast
      enough to actually run.

      Notice that "Row 0: try col 0" leads to a huge amount of dead-end

      entirely and backing all the way up to row 0. That's normal. Backtracking
      doesn't avoid dead ends, it just abandons them the moment they're
      detected (as soon as a row runs out of safe columns), instead of only
      after wastefully filling in the rest of the board first.
