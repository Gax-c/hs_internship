Introduction to Verified Programming

Today's goal: understand why "it looks right" isn't the same as "it is right" — and prove it to yourself by breaking some code.

Learn

📖 Code, Tests, Specs, and Proofs

Imagine you write a function that checks if a year is a leap year. You run it on the year 2024, and it says "yes." Great — but does that mean your code is correct? Not really. It means your code gives the right answer for one input. There could be a thousand other inputs where it's wrong.

This week is about closing that gap, using four ideas that build on each other: code, tests, specifications, and proofs. We'll use a language called Rust all week, and on Day 3 a real verification tool called Verus. Let's take things one at a time, starting from the very beginning — assume nothing, we'll build it all up together.

What does code actually look like?

Code is just a very precise, very literal set of instructions for a computer — more like a recipe than an essay. Here's a handful of the building blocks you'll see everywhere this week, in Rust:

TermWhat it meansWhat it looks like
VariableA labeled box that holds a valuelet age = 16;
FunctionA reusable recipe: takes some input, does steps, hands back an outputfn double(x: i32) -> i32 {
    x * 2
}
TypeWhat kind of value something is — Rust always wants to knowi32 (a whole number), bool (true/false)
OperatorA symbol that does an operation — math, or a comparison+ - * / % and == != && ||
ConditionCode that only runs when something is trueif age >= 18 { ... }
Return valueThe answer a function hands back — in Rust, usually just the last line, with no semicolonx * 2
CommentA note for humans; the computer ignores anything after //// this is a note

One operator worth calling out early, since it's the heart of the leap-year rule: % ("modulo") gives you the remainder after division. 10 % 3 is 1, because 10 divided by 3 is 3 with 1 left over. "year % 4 == 0" is just code for "year divides evenly by 4."

What is a bug?

    

      A bug is a mistake in code — a place where the code does
      something different from what the person writing it actually meant. The tricky
      part: buggy code usually doesn't crash or show an error message. It just quietly
      gives a wrong answer, and looks completely normal while doing it. That's what
      makes bugs hard to catch just by reading code — you often have to actually
      run it on the right input to notice anything is wrong.
    

    

      Where the name comes from: engineers were already calling
      mechanical glitches "bugs" in the 1800s, but computing's most famous one is real:
      in 1947, a team working on the Harvard Mark II computer found an actual moth
      trapped in one of its relays, causing a malfunction. They taped it into their
      logbook, labeled "first actual case of bug being found." The name stuck.

What does a specification look like?

    

      Let's use the smallest example possible: a function that checks whether a number
      is even. Someone asks you for this in plain English:
    

    
Write a function that checks if a number is even.

    

      That sounds obvious — but before you write a single line of code, ask
      yourself some questions:
    

    
    
      
  • What should happen for 0? (Is zero even? Yes — but would you have been sure without asking?)
    • 
      
  • What about a negative number, like -4?
    • 
      
  • What about a number that isn't a whole number, like 2.5? The description never said "whole number" — though in Rust, choosing the type i32 (whole numbers only) already answers this question for us before we even get to write the rule!
    • 
    

    

      A specification is what you get once you've answered every one of
      those questions and written the rule down precisely, so there's no room left for
      "wait, what should it do here?" Here's one:
    

    

      Specification: is_even(n)
      
    
        
  • Input: n is a whole number, type i32 (positive, negative, or zero).
    • 
        
  • Output: true if n divided by 2 leaves no remainder, otherwise false.
    • 
        
  • Decided edge cases: 0 counts as even. Negative numbers follow the same rule as positive ones (-4 is even, -3 is odd).
    •

How the four ideas fit together

    

      

        

          
1. Code

          
The program that's supposed to solve the problem. It might have bugs.

        

        
→

        

          
2. Tests

          
Specific examples: "for this input, the answer should be that." Catches known problem cases.

        

        
→

        

          
3. Specification

          
A precise rule for every possible input, not just examples. Removes ambiguity.

        

        
→

        

          
4. Proof

          
A mathematical guarantee the code matches the spec for every input, forever.

        

      

    


    
What is a proof, really?

    

      Suppose you want to know: is the sum of any two even numbers always even?
      Let's check a few:
    

    
    
      
  • 2 + 4 = 6 (even)
    • 
      
  • 6 + 8 = 14 (even)
    • 
      
  • 10 + 100 = 110 (even)
    • 
    

    

      Every test passes so far. But we've only checked 3 pairs out of infinitely many
      possible pairs of even numbers — no matter how many more we try, we can
      never test all of them by hand. A proof settles the
      question completely, using logic instead of examples. Here's the whole argument,
      one step at a time:
    

    
    
      
  1. Every even number is "2 times some whole number." For example, 6 = 2 × 3 and 10 = 2 × 5.
    2. 
      
  2. So let's give the two whole numbers names: say our first even number is 2 × a, and our second is 2 × b — where a and b could be any whole numbers at all.
    3. 
      
  3. Their sum is (2 × a) + (2 × b), which is the same thing as 2 × (a + b).
    4. 
      
  4. "2 times (a + b)" is even, by the very definition of what "even" means — no matter what number a + b turns out to be.
    5. 
    

    

      Check it against the examples above: for 6 + 10, we have a = 3 and b = 5, so the proof says the sum is 2 × (3 + 5) = 2 × 8 = 16. Sure enough, 6 + 10 = 16, and it's even — exactly as the general argument predicted, without us having to test that specific pair. Because a and b stood for any whole numbers, this one argument covers every possible pair of even numbers there will ever be. You've likely written a proof shaped like this in a math class already.
    

    

      A proof about code works the same way: instead of running the program on a
      thousand inputs and hoping you covered the important ones, a proof reasons about
      all inputs at once and guarantees the code matches its specification
      forever — not just for the cases someone happened to try. This isn't just a
      nice idea for this internship: on Day 3, you will use a real tool
      called Verus to write exactly this kind of proof about your own
      is_leap function, and it will tell you, for certain, whether it holds
      for literally every possible input.
    


    

      Bugs like these have caused real, expensive disasters — a rocket
      (Ariane 5) self-destructed because of a single unhandled number overflow, and a
      radiation therapy machine (Therac-25) gave lethal overdoses because of a race
      condition no one tested for. Nobody thought their code was wrong. Nobody had
      tested the specific case that broke it. Nobody had proved it correct, either
      — if they had, that overflow would have been caught before it ever flew.
    


    

      VeriContest is a research effort (the reason this internship
      exists) that studies exactly this: given a problem, can we write down a precise
      specification, and can we automatically check — or even prove — that
      code, including code written by an AI, actually satisfies it? This week you'll
      build a tiny version of that yourself, called Mini-Judge, using
      real tools (Rust and Verus) rather than a simplified stand-in. See
      The Project for the full picture.
