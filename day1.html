📖 Code, Tests, Specs, and Proofs

Why Code Correctness Matters

Software runs almost everything today: rockets, hospital equipment, airplanes, banks, the app you used this morning. When code looks like it works, it's easy to assume it is correct. But "looks right" and "is right" are different claims, and the gap between them has caused some of the most expensive and dangerous failures in computing history. Here are six real ones. Each links to a fuller writeup if you want the details:

☢️ Therac-25, 1985–1987

A radiation therapy machine had a bug that only appeared if an operator entered treatment settings in one order and then corrected them within a specific, narrow timing window: an edge case nobody happened to test. When it triggered, patients received radiation doses far beyond what was intended. Several people died.

Learn more about Therac-25 →

🚀 Ariane 5, 1996

A European rocket veered off course and self-destructed 37 seconds after launch, destroying itself and its satellite payload for a loss of about $370 million. The cause: leftover navigation software from an earlier, slower rocket tried to convert a speed measurement into a 16-bit number. On the new, faster rocket, the number was too big to fit. Nobody re-tested that old code against the new rocket's actual numbers.

Learn more about Ariane 5 Flight 501 →

🛰️ Mars Climate Orbiter, 1999

NASA lost a $327.6 million spacecraft because one engineering team's software calculated thruster force in pound-force-seconds, while the team using that number expected newton-seconds, off by a factor of 4.45. Every line of code ran exactly as written; the two teams had simply never agreed on precisely what their numbers meant.

Learn more about Mars Climate Orbiter →

📅 Y2K, 1999–2000

For decades, programs commonly stored years using only two digits (99 instead of 1999), so many systems risked reading the year 2000 as 1900. This one is different from the rest: the bug was found before it caused a disaster, and the world spent an estimated $300–600 billion rewriting old code in advance. Correctness work that worked looks like nothing happened, which is exactly the point.

Learn more about the Y2K problem →

📈 Knight Capital, 2012

A deployment script updated 7 of 8 trading servers with new code, but silently failed on the 8th and reported success anyway. That one server still had a dormant feature from 2003, which a reused signal accidentally switched back on. In 45 minutes it fired 4 million unwanted trades, losing $440 million and nearly bankrupting the company.

Learn more about Knight Capital →

💻 CrowdStrike, 2024

A single security-software update expected 20 pieces of data but was given 21, causing computers to read memory that wasn't there and crash. About 8.5 million Windows computers went down at once, grounding over 10,000 flights and costing an estimated $5.4 billion. One update, pushed to everyone at the same time, with no case like it caught beforehand.

Learn more about the CrowdStrike outage →

Notice these have different root causes: an untested edge case, an unexamined assumption carried over from old code, two teams disagreeing on what a number even means, an update shipped to everyone before anything like it was checked, a silent deployment failure nobody noticed, and (the one exception) a problem found and fixed before it could cause any of this. That's exactly why this week isn't about just one fix. You'll build four different tools this week (code, tests, specifications, and proofs) because each one catches a different kind of mistake, and real software needs more than one line of defense.

Here's the same idea at a much smaller, safer scale. Imagine you write a function that checks if a year is a leap year. You run it on the year 2024, and it says "yes." Great, but does that mean your code is correct? Not really. It means your code gives the right answer for one input. There could be a thousand other inputs where it's wrong. That's the same basic gap behind every story above, just with much lower stakes.

This week is about closing that gap, using the four ideas you just read about: code, tests, specifications, and proofs. We'll use a language called Rust for most of the week, and on Day 4 a real verification tool called Verus. Let's take things one at a time, starting from the very beginning: assume nothing, we'll build it all up together.

What does code actually look like?

Code is just a very precise, very literal set of instructions for a computer, more like a recipe than an essay. Here's a handful of the building blocks you'll see everywhere this week, in Rust:

TermWhat it meansWhat it looks like
VariableA labeled box that holds a valuelet age = 16;
FunctionA reusable recipe: takes some input, does steps, hands back an outputfn double(x: i32) -> i32 {
    x * 2
}
TypeWhat kind of value something is (Rust always wants to know)i32 (a whole number), bool (true/false)
OperatorA symbol that does an operation: math, or a comparison+ - * / % and == != && ||
ConditionCode that only runs when something is trueif age >= 18 { ... }
Return valueThe answer a function hands back (in Rust, usually just the last line, with no semicolon)x * 2
CommentA note for humans; the computer ignores anything after //// this is a note

One operator worth calling out early, since it's the heart of the leap-year rule: % ("modulo") gives you the remainder after division. 10 % 3 is 1, because 10 divided by 3 is 3 with 1 left over. "year % 4 == 0" is just code for "year divides evenly by 4."

What is a bug?

    

      A bug is a mistake in code: a place where the code does
      something different from what the person writing it actually meant. The tricky
      part: buggy code usually doesn't crash or show an error message. It just quietly
      gives a wrong answer, and looks completely normal while doing it. That's what
      makes bugs hard to catch just by reading code. You often have to actually
      run it on the right input to notice anything is wrong.
    

    

      Where the name comes from: engineers were already calling
      mechanical glitches "bugs" in the 1800s, but computing's most famous one is real:
      in 1947, a team working on the Harvard Mark II computer found an actual moth
      trapped in one of its relays, causing a malfunction. They taped it into their
      logbook, labeled "first actual case of bug being found." The name stuck.

What does a specification look like?

    

      Let's use the smallest example possible: a function that checks whether a number
      is even. Someone asks you for this in plain English:
    

    
Write a function that checks if a number is even.

    

      That sounds obvious, but before you write a single line of code, ask
      yourself some questions:
    

    
    
      
  • What should happen for 0? (Is zero even? Yes, but would you have been sure without asking?)
    • 
      
  • What about a negative number, like -4?
    • 
      
  • What about a number that isn't a whole number, like 2.5? The description never said "whole number," though in Rust, choosing the type i32 (whole numbers only) already answers this question for us before we even get to write the rule!
    • 
    

    

      A specification is what removes that ambiguity. In
      this internship, "specification" means something specific: not a paragraph of
      English, but real code that a computer can check automatically. In Verus, you
      write a specification using a special kind of function called a
      spec fn. Unlike ordinary code, a spec fn is never
      actually run. It exists purely as a precise mathematical statement for
      Verus to reason about. Here's the leap-year problem's cousin, written as one:

How the pieces fit together: two workflows

    

      Notice "code" showed up in both of the last two examples, but "specification"
      only showed up in the second one, and in a different position each time.
      That's not an accident: there are really two different
      workflows for turning a vague description into something you can
      trust, and this week you'll do both.
    


    

      
Workflow 1: Testing (Day 2)

Specific input/output examples. Passing them doesn't rule out a bug hiding in a case nobody tried.

        

      


      
Workflow 2: Formal Verification (Day 4)

      

        

          
1. Description

          
The same kind of starting point: a natural-language ask.

        

        
→

        

          
2. Formal Spec

          
A precise rule for every input, written as code (a spec fn), not English.

        

        
→

        

          
3. Code

          
An implementation, written to match the specification.

        

        
→

        

          
4. Proof

          
A guarantee, checked by Verus, that the code matches the spec for every input, forever.

        

      

    


    

      Days 2 and 3 are workflow 1: you'll code (with AI assistance) and test it. The
      leap-year bug hunt today is a preview of exactly this. Day 4 takes it further
      with workflow 2: you'll write a formal specification first, then get
      Verus to prove your code matches it, for every input rather than just
      the ones you happened to try.
    


    
What is a proof, really?

    

      Suppose you want to know: is the sum of any two even numbers always even?
      Let's check a few:
    

    
    
      
  • 2 + 4 = 6 (even)
    • 
      
  • 6 + 8 = 14 (even)
    • 
      
  • 10 + 100 = 110 (even)
    • 
    

    

      Every test passes so far. But we've only checked 3 pairs out of infinitely many
      possible pairs of even numbers. No matter how many more we try, we can
      never test all of them by hand. A proof settles the
      question completely, using logic instead of examples. Here's the whole argument,
      one step at a time:
    

    
    
      
  1. Every even number is "2 times some whole number." For example, 6 = 2 × 3 and 10 = 2 × 5.
    2. 
      
  2. So let's give the two whole numbers names: say our first even number is 2 × a, and our second is 2 × b, where a and b could be any whole numbers at all.
    3. 
      
  3. Their sum is (2 × a) + (2 × b), which is the same thing as 2 × (a + b).
    4. 
      
  4. "2 times (a + b)" is even, by the very definition of what "even" means, no matter what number a + b turns out to be.
    5. 
    

    

      Check it against the examples above: for 6 + 10, we have a = 3 and b = 5, so the proof says the sum is 2 × (3 + 5) = 2 × 8 = 16. Sure enough, 6 + 10 = 16, and it's even, exactly as the general argument predicted, without us having to test that specific pair. Because a and b stood for any whole numbers, this one argument covers every possible pair of even numbers there will ever be. You've likely written a proof shaped like this in a math class already.
    

    

      That even-plus-even argument was simple enough that, in Verus, you can state it
      and the SMT solver just figures out the whole thing by itself, with no help from
      you. That won't always be true. Here's a small function with a loop
      where Verus genuinely needs your help, in the form of a loop
      invariant: a fact you tell it that stays true on every single pass
      through the loop.

      Click Run: it verifies. This function counts up from 0 to n one
      step at a time, and ensures result == n promises it always ends up
      exactly at n. Verus can't see that by itself. It has to reason
      about the loop somehow, and the invariant i <= n line is your
      hint: "no matter how many times this loop has run, i is never
      bigger than n." Combine that with the loop only stopping once
      i < n is false, and it follows that i must equal
      n exactly. Try deleting the whole invariant i <= n,
      line and clicking Run again. Verus will refuse it, because without that hint it
      has no way to connect what the loop does to what ensures promises.
      (The decreases line is a second, separate proof obligation: that
      the loop is guaranteed to stop at all.)
    

    

      A proof about code works the same way: instead of running the program on a
      thousand inputs and hoping you covered the important ones, a proof reasons about
      all inputs at once and guarantees the code matches its specification
      forever, not just for the cases someone happened to try. On Day 4, you will use a real tool
      called Verus to write exactly this kind of proof about your own
      is_leap function, and it will tell you, for certain, whether it holds
      for literally every possible input.
    


    

      Think back to Ariane 5 and Therac-25 from the start of today. A proof is exactly
      the kind of tool that could have caught both, not by getting lucky on the
      right test case, but by checking every possible case at once, before the code
      ever flew or treated a patient. Nobody had proved either one correct. If they
      had, the overflow and the race condition would both have been caught in
      advance, not discovered by the failure itself.
    


    

      VeriContest is a research effort (the reason this internship
      exists) that studies exactly this: given a problem, can we write down a precise
      specification, and can we automatically check, or even prove, that
      code, including code written by an AI, actually satisfies it? This week you'll
      build a tiny version of that yourself, called VeriCode, using
      real tools (Rust and Verus) rather than a simplified stand-in. See
      The Project for the full picture.
