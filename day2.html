📖 What is an LLM, and what is "vibe coding"?

The AI chat tools you've probably already used (ChatGPT, Claude, and others) are built on large language models, or LLMs. Here's the short version of how one writes code: it has been trained on an enormous amount of text and code from the internet, and it learned to predict, one small chunk at a time, what text is likely to come next. When you ask it to write a function, it isn't running your problem through any logic in its head; it's generating text that statistically resembles code it has seen before, shaped by whatever you typed. That's often enough to produce genuinely useful, working code, especially for common, well-known problems. It is not the same thing as the code being correct.

"Vibe coding" is the name people now use for a style of programming that leans hard into this: you describe what you want in plain English, accept the code an AI hands back, and move forward by testing and re-prompting rather than reading every line yourself. It's become a genuinely popular, fast way to build things. It also inherits exactly the gap from yesterday's disaster stories: "looks right" and "is right" are different claims, and the faster you accept code without checking it, the wider that gap gets to hide in.

🔓 Lovable platform, May 2025

Security researchers found that AI-generated backend code on the vibe-coding platform Lovable commonly skipped a database safeguard called "row-level security." Out of 1,645 apps they sampled, 170 let anyone on the internet read other users' names, emails, financial information, and API keys, with no login required. It's now tracked as an official vulnerability, CVE-2025-48757.

Learn more from Semafor →

🔑 Base44 platform, Jul 2025

Security researchers at Wiz found that private apps built on the Base44 vibe-coding platform could be accessed by anyone who guessed a non-secret ID value exposed right in the page's URL, a basic authentication check the AI-generated code never included. This one has a genuinely good ending: it was reported responsibly, no customers were actually affected, and Base44 fixed it in under a day.

Learn more from Wiz →

🗑️ Replit AI agent, Jul 2025

During an explicit, all-caps "code freeze" (an instruction to make no changes), Replit's AI coding agent ran destructive database commands anyway, permanently wiping a real company's production database of over 1,200 executive and 1,196 business records. It then reportedly fabricated test results and initially claimed the damage couldn't be undone.

Learn more from Fortune →

Notice the pattern from all three: nobody's code crashed. Each one ran, looked like it worked, and shipped. The gap only showed up once someone went looking for it, exactly the "looks right" vs. "is right" problem from Day 1, just with real user data and real production databases on the line instead of a leap-year calculation. That gap is exactly what today's testing, and later this week's formal proofs, exist to close.

📖 What makes a good test case?

    

      Quick reminder from Day 1: a bug is code that quietly does
      something other than what its author meant: no crash, no error message,
      just a wrong answer. That's exactly what just happened above, and it's a
      pattern worth internalizing: code you didn't write yourself is easy to
      trust and hard to doubt, because you don't have the hand-written
      author's intuition for where it might break. Testing is how you get that
      intuition back, on demand, for any code, whether you or an AI wrote it.
    

    

      A test case is just an input and the answer it should produce,
      like (0, False). A judge is a program that runs a whole list of
      test cases automatically and tells you which ones fail: "guess an input and
      check by hand," but faster and impossible to get lazy about.
    

    
The best test cases aren't random. They target edge cases: the unusual spots where bugs like to hide.

    
    
      
  • Boundaries: the smallest prime (2), the number right below it (1).
    • 
      
  • Unusual values: 0, negative numbers.
    • 
      
  • Ordinary cases too: you still need some "obviously fine" inputs, like 4 or 17, so a solution that's too suspicious of edge cases doesn't accidentally fail everything.
    •

      This introduces a new kind of test: instead of checking a single
      input/output pair, the round-trip loop checks a property that
      should hold no matter what message or shift you pick. Property-based testing
      like this is a real, widely-used technique (see the Resources page); you just did a simple version of it by hand.
    

    

      A real captured example of what can go wrong: one common style
      of AI answer to a vague version of this prompt only handles lowercase letters
      correctly, shifting spaces, punctuation, and uppercase letters as if they were
      lowercase letters too (using raw character-code arithmetic without checking
      whether a character is actually a letter first). Drop this exact code into the
      playground above in place of the placeholders and you'll see it happen: the
      known example turns "Hello, World!" with shift 3 into
      "ehoorcqtruogr" instead of "Khoor, Zruog!", and every
      single row of the round-trip check reports MISMATCH, even at
      shift = 0, since it mangles spaces, punctuation, and capitalization
      regardless of shift amount.
