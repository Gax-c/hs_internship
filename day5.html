Proofs, Both Workflows & Wrap-Up

Today has two parts. First, Verus proofs: yesterday you wrote specs, the promises code has to keep; today you learn the tools for actually convincing Verus a promise is kept, for the cases where its automatic solver can't see it alone. Second, the final report: pulling this whole week (testing and formal verification) into one write-up you can print or save as a PDF.

Learn · Part 1: Verus Proofs

📖 Why do we need proofs?

All week, connecting a spec to code has looked the same: write ensures result == my_spec(...), and Verus's automatic solver (Z3) works out on its own whether the code lives up to it. That's exactly what happened for is_leap, letter grades, triangles, and even the real VeriContest Nim game problem: Z3 handled the whole thing itself, with zero extra work from you.

But Z3 isn't magic, and it has real, specific blind spots. Three of them matter for today:

  • Multiplication of two unknown values ("nonlinear arithmetic") is notoriously hard for automatic solvers, even for facts a human finds obvious.
  • Loops can run an unknown number of times, so Z3 can't just "try every iteration" the way it tries every possible input to a spec.
  • Recursive specs (like the ones in yesterday's VeriContest examples) could unfold forever if Z3 tried to expand them blindly, so it deliberately doesn't, unless told to.

When Z3 hits one of these, verification just fails, the same postcondition not satisfied style error you've seen all week, even though the code is genuinely correct. A proof is how you close that gap: extra reasoning, written in the same file, that walks Z3 through exactly the step it couldn't find alone. You saw several already, pre-written, inside yesterday's VeriContest exercises; today you write your own. There are three tools for this, and you'll use all three today: assert, loop invariant, and lemma functions (proof fn).

Learn

📖 Proof Type 1: assert

The simplest proof tool is also the one you already half-know: assert(fact) tells Verus "prove this specific fact is true, right here." It's a checkpoint, not a whole argument: you're breaking a big claim into a smaller, more manageable one, at one exact spot in the code. If the fact is actually true (and Verus can see why), it moves on silently. If it isn't, or Verus can't see why, you get an error pointing at that exact line.

      invariant sitting in the code, already written, so you could focus on
      the spec. Today you write one yourself.

      specific moment, repeated over and over: right before Verus checks the
      loop's condition, every single time, including before the very first
      iteration and right after the very last one. That repetition is exactly
      why it works without "unrolling" the loop: Verus only ever has to check
      one thing, "does this iteration preserve the invariant," instead of
      reasoning about every possible number of iterations separately.

📖 Proof Type 3: lemma functions (proof fn)

    

      You've also already seen this one: yesterday's "steps to zero" problem
      had two small proof fns underneath the spec, already
      written. A lemma function is a function that contains
      only logical reasoning, never actual running code, whose entire
      purpose is establishing one reusable fact. It's especially useful for
      recursive specs, where the natural way to prove something is to mirror
      the recursion itself: prove the base case, then show each recursive
      case follows from a smaller version of the same fact, exactly the
      structure of a mathematical proof by induction.
    

    

      Here's a fact about even_count_spec from the invariant
      example above: it never returns more than n itself.

      Read it the way you'd read an induction proof by hand: "base case,
      n <= 0: even_count_spec is 0, and 0 <= 0,
      done. Recursive case, n > 0: assuming the fact already
      holds for n - 1 (that's what calling
      even_count_bound(n - 1) gets you), the fact for
      n follows." decreases n is required here for
      the same reason it was required on recursive spec fns
      yesterday: proving each call genuinely gets closer to the base case, so
      the reasoning can't loop forever.
    

    

      A lemma only helps once you actually call it. Proved on its
      own, even_count_bound doesn't automatically make anything
      else aware of the fact; you have to invoke it, inside a
      proof { } block, at the point where you need it, the same
      way yesterday's "steps to zero" proof called its two lemmas from inside
      the loop.

📖 How do you actually write a proof?

    

      Now that you've used all three tools separately, here's the honest
      answer to "how do I know which ones I need, and when?" Part of that
      answer is a hard rule. The rest of it, you only ever learn by doing.
    


    
The one rule: a loop always needs an invariant

    

      If your executable code has a loop, and that loop's result has to match
      a spec through ensures, you always need a
      loop invariant. Full stop. This part isn't a judgment call the way
      asserts and lemmas are: Verus fundamentally cannot reason past a loop
      without one, the same way it fundamentally cannot verify a function with
      no ensures at all. Even a loop that obviously does the right
      thing still needs its invariant spelled out; Verus never runs your code
      to see what "obviously" happens, it only ever reads what you've told it.
    


    
Asserts and lemmas: it depends on the program

    

      assert and lemma functions are a different story. Compare
      the two loop-based problems you've now seen in full. Today's
      count_evens/count_odds needed
      only an invariant: no assert, no lemma, nothing else.
      But yesterday's "steps to zero" problem needed an invariant
      and two separate lemma functions underneath it. Both problems
      have a loop. Only one of them needed help beyond the invariant. That
      difference wasn't a rule you could have looked up ahead of time; it came
      entirely from what that specific loop's invariant needed to lean on.
    

    

      That's the real answer: whether a given program needs an
      assert or a lemma is empirical. It depends
      on the exact arithmetic and logic in that specific program, not on some
      general property of "loops" or "specs" you could memorize once and reuse
      everywhere. Nobody, including people who write Verus proofs
      professionally, reliably predicts this from just reading code; they find
      out by trying it.
    


    
The actual workflow

    
This is genuinely how it's done in practice, including by whoever wrote every VeriContest proof you've seen this week:

    
    
      
  1. Write your executable code and its spec. If there's a loop, write your best-guess invariant, the strongest true statement you can come up with that captures what the loop is doing. Add nothing else yet.
    2. 
      
  2. Run Verus.
    3. 
      
  3. 
        If it verifies, stop. You didn't need an assert or a
        lemma this time, and adding one anyway would just be unnecessary
        extra work. This is exactly what happened with is_leap,
        letter grades, triangles, the Nim game, and count_evens.
      
    4. 
      
  4. 
        If it doesn't, read the specific error. Verus tells
        you exactly which line and which fact it couldn't establish, the same
        way it's done all week. That failure is your instruction for what to
        add next, not a sign to throw everything out and start over.
      
    5. 
      
  5. 
        Add the smallest hint that addresses that exact gap, and nothing more:
        an assert(...) by (nonlinear_arith) if the failure
        involves multiplying two unknowns, a lemma call if it involves a fact
        about a recursive spec, or a stronger invariant clause if the loop's
        own reasoning was too weak. Run again.
      
    6. 
      
  6. Repeat. Most real proofs, including yesterday's VeriContest examples, were built exactly this way: a handful of rounds of "run it, read the error, add the smallest fix that addresses it," not written correctly in one pass.
    7. 
    


    
Two rough patterns worth noticing, even though neither is a guarantee:

    
    
      
  • Multiplying or dividing two unknown values, in a fact Verus needs to see, is a strong hint you'll want an assert(...) by (nonlinear_arith), like today's square_at_least_self.
    • 
      
  • A recursive spec your loop invariant depends on (like even_count_spec or yesterday's spec_steps) is a strong hint you'll eventually want a lemma proving some fact about it by induction, especially a numeric bound needed to avoid overflow.
    • 
    

    

      Neither pattern is a promise, only something worth noticing once you've
      seen it happen a few times. The only way to know for certain, for any
      specific piece of code, is the workflow above: try it, and let Verus's
      own error message tell you what's actually still missing.
