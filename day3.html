Learning Rust

Most of today is not about writing a Rust program from a blank page: it's about being able to look at a Rust program and understand what it's doing. Every exercise gives you a program that's almost right, with one small thing broken; your job is to find it, understand why it's wrong, and fix it. Near the end, once that's second nature, you'll get a chance to write some small Rust of your own, with as much or as little help as you want.

Learn

📖 Variables, and a very Rust idea: mut

A variable is just a name a program uses to refer to a value, so you don't have to keep repeating that value everywhere. Instead of writing the number 17 five times in your code, you give it a name once (say, n) and then use n everywhere you mean "that value." In Python you wrote n = 17 to create one. In Rust, you write let n = 17;: the word let is how Rust says "create a new variable," and the semicolon marks the end of that instruction.

Every value in Rust has a type: a category that says what kind of thing it is and what you're allowed to do with it. Rust can usually figure out the type automatically just from the value you wrote (this is called type inference), but you can also spell it out explicitly with a colon, like let n: i32 = 17;. That i32 is the name of a type, and it's worth knowing exactly what it means, since you'll see it constantly this week:

  • The i stands for integer: a whole number, with no decimal point, that can be negative, zero, or positive.
  • The 32 is how many bits of computer memory are set aside to store it. A bit is the smallest unit of memory a computer has: it's like a tiny switch that can only be off (0) or on (1). Using 32 of these switches together, a computer can represent a little over 4.3 billion distinct values.
  • Split between negative and positive numbers, that gives i32 a range of about −2.1 billion to +2.1 billion. A number bigger or smaller than that simply doesn't fit in 32 bits; you'll see this cause a real, genuine problem later this week, not just a hypothetical one.

This is different from Python, where whole numbers can be as large as your computer's memory allows, growing automatically as needed. Rust asks you to pick a fixed size up front (there are other integer types too, like the smaller i8 or the larger i64, but i32 is the everyday default we'll use all week) because fixed-size numbers are what computer hardware works with directly, which makes Rust programs fast and lets the compiler double-check your arithmetic before the program ever runs.

Here's the genuinely new idea for today: in Rust, a variable can't be changed after it's created, unless you say so. Write let mut counter = 0; (mut is short for mutable, meaning "changeable") if you plan to update it later. Leave out mut, and Rust treats that variable as fixed forever after its first value: this is called immutable. This is different from Python or most other languages you might have heard of, where every variable can always be reassigned at any time. Rust's design philosophy is to catch mistakes early: if you never intended to change a variable, and your code accidentally tries to anyway (a typo, or a line you forgot you wrote), Rust refuses to even compile the program, instead of letting that mistake slip through silently.

📖 The basic types you'll actually use

    
Besides i32, there are three more basic types worth knowing exactly:

    
    
      
  • f64: a number that's allowed to have digits after a decimal point, like 3.9. The f stands for floating-point (the decimal point can "float" to different positions, which is how the same 64 bits can represent both very large and very tiny decimal values), and the 64 is again how many bits of memory it uses: more than i32's 32, since decimal numbers need extra room to store the digits after the point.
    • 
      
  • char: exactly one single character, like a letter, digit, or punctuation mark, always written in single quotes: 'A'. This is different from text made of many characters, which Rust calls a String and writes in double quotes, like "Ada".
    • 
      
  • bool: short for boolean, named after the mathematician George Boole. A bool can only ever be exactly one of two values: true or false. You've actually been reading this type constantly since Day 1: is_leap always returns a bool.

📖 Functions: types are required, and the last line is the answer

    

      A function is a named, reusable block of instructions. You've
      used dozens of them already this week: is_leap, is_prime,
      and judge are all functions. A function can accept input values,
      called parameters, and produce an output value, called its
      return value. In Python you'd write
      def add(a, b): return a + b: def starts the
      definition, a and b are parameters with no type
      attached, and return explicitly says what comes back.
    

    

      Rust's version looks like fn add(a: i32, b: i32) -> i32 { a + b },
      and every piece of it is required, not optional:
    

    
    
      
  • fn is short for function, the same role as Python's def.
    • 
      
  • Each parameter needs both a name and a type, separated by a colon: a: i32 means "a parameter named a, which must be an i32." Rust insists on this so it can check, before your program ever runs, that every value flowing through your code is the kind of value it's supposed to be.
    • 
      
  • -> i32 (read "returns i32") declares the type of the answer the function produces. This has to be written even if it seems obvious.
    • 
      
  • Notice there's no return keyword and no semicolon on a + b. In Rust, if the very last line of a function has no semicolon, its value automatically becomes the function's return value. A semicolon marks the end of an ordinary instruction that doesn't hand back a value; leaving it off on the last line is what makes that line the answer.

📖 Operators

    
An operator is a symbol that combines values to produce a new one, like + or >. Rust's arithmetic operators are the ones you already know: + add, - subtract, * multiply, / divide, and % "modulo," meaning the remainder left over after dividing (you've been reading % all week in year % 4). Its comparison operators produce a bool: == equal, != not equal, <, >, <=, >=. Combining true/false values is spelled differently than in Python: && instead of and (true only if both sides are true), || instead of or (true if at least one side is true), and ! instead of not (flips true to false and back).

    
One real surprise if you're coming from Python: dividing two whole numbers with / in Rust always gives back a whole number, throwing away anything after the decimal point, rather than automatically becoming a decimal like Python's / does. 7 / 2 is 3, not 3.5. Watch for this below.

📖 if/else is an expression, and loops

    

      An if/else lets a program choose between two
      different paths depending on whether a condition is true or
      false. The shape looks a lot like Python's, but with two syntax
      differences: Rust doesn't need parentheses around the condition, and it always
      needs curly braces { } around each branch's body (Python uses a
      colon and indentation instead).
    

    

      Here's a genuinely different idea from Python, though: in Rust,
      if/else itself produces a value, the same way a
      function's last line does. In Python, if/else just
      chooses which instructions to run; it isn't something you can save into a
      variable. In Rust, the whole if/else block evaluates
      to whichever branch actually ran, so you can hand it directly back as a
      function's answer, or store it straight into a variable, with no separate
      return needed. The function below is supposed to compute absolute
      value (always a non-negative result: turn any negative number positive, and

📖 A closer look at println!, and a peek at &

    

      You've used println! in every single Rust program so far. The
      ! at the end means it's a macro, not an ordinary
      function: macros are allowed to do things regular functions can't, like accept
      a different number of arguments each time you call them. Inside the quoted
      text, each {} is a placeholder, filled in order by the arguments
      listed after the comma. The number of {} placeholders has to
      exactly match the number of arguments you provide. Fix the line below so it

      One more symbol you'll see starting tomorrow, worth recognizing even though we
      won't dwell on it today: an & in front of a value, like in
      &test_cases or the type &[(i32, bool)] in your
      Day 2 judge skeleton. In plain terms, & means "let this code
      look at the value without taking it away permanently": the value still belongs
      to wherever it came from, and this is just borrowing a look at it. The full
      story behind & (Rust calls it a reference) is
      one of Rust's more advanced ideas, and you don't need to master it today; just
      recognize it and know roughly what it's for when you see it.
